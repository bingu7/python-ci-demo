## 1. CI/CD 基本概念

### CI（Continuous Integration，持续集成）

核心问题：多个人写代码，每人写一部分，怎么保证合在一起还能跑？

解决方式：每次提交代码，自动跑测试，确保没有破坏现有功能。

```
提交代码 → 自动构建 → 自动跑测试 → 报告结果
```

"持续"的意思：不是等项目做完了才测，而是每次改动都测。

CI 做的事：
1. 拉取最新代码
2. 安装依赖
3. 编译（如果需要）
4. 跑单元测试
5. 跑集成测试
6. 代码质量检查（代码风格、安全漏洞等）

### CD（Continuous Delivery / Deployment）

**持续交付（Continuous Delivery）：**
```
CI 测试通过 → 自动打包 → 准备好可以部署 → 人工点一下部署
```
测试通过后，软件已经可以随时发布，但发布还需要人工确认。

**持续部署（Continuous Deployment）：**
```
CI 测试通过 → 自动打包 → 自动部署到生产环境
```
全程自动化，不需要人工干预。

### 完整流程图

```
开发者写代码
    ↓
git push 到远程仓库（GitHub）
    ↓
 CI 触发（GitHub Actions 自动运行）
    ├→ 安装环境
    ├→ 跑单元测试
    ├→ 跑集成测试
    ├→ 代码检查
    └→ 安全扫描
    ↓
 全部通过？
    ├→ ✅ 通过 → 自动部署（CD）
    └→ ❌ 失败 → 通知开发者修复
```

### 为什么重要

1. 早发现 Bug — 写完就测，不是等上线了才发现
2. 减少手动工作 — 不用手动跑测试、手动部署
3. 保证代码质量 — 合并前必须测试通过
4. 团队协作安全 — 你的代码不会搞坏别人的代码
5. 随时可发布 — main 分支永远是可部署的状态

## 2. Git 常用命令

### 仓库初始化

```bash
git init                      # 初始化新仓库
git clone <url>               # 克隆远程仓库
```

### 日常工作流（最核心）

```bash
git status                    # 查看当前状态
git add <文件名>              # 添加指定文件到暂存区
git add .                     # 添加所有改动
git commit -m "描述"          # 提交到本地仓库
git push                      # 推送到远程仓库
git push -u origin main       # 第一次推送，建立关联
```

工作流：改代码 → add → commit → push

### 分支操作

```bash
git branch                    # 查看所有分支
git branch <分支名>           # 创建新分支
git checkout <分支名>         # 切换分支
git checkout -b <分支名>      # 创建并切换（合并写法）
git branch -d <分支名>        # 删除分支
```

### 合并与拉取

```bash
git merge <分支名>            # 合并指定分支到当前分支
git pull                      # 拉取远程最新代码并合并（= fetch + merge）
git fetch                     # 只拉取，不合并
```

### 查看历史

```bash
git log                       # 详细提交历史
git log --oneline             # 简洁版，每条一行
git log --oneline -10         # 只看最近 10 条
git diff                      # 工作区 vs 暂存区
git diff --staged             # 暂存区 vs 最后一次 commit
```

### 撤销操作

```bash
git restore --staged <文件>   # 取消暂存（撤回 add）
git restore <文件>            # 放弃工作区修改（危险）
git commit --amend -m "新描述" # 修改最后一次 commit 信息
git reset --hard <commit-id>  # 回退到某 commit（丢弃所有改动，危险）
```

### 远程仓库

```bash
git remote -v                 # 查看远程地址
git remote add origin <url>   # 添加远程仓库
git remote set-url origin <url> # 修改远程地址
```

### 完整工作流示例（修复 Bug）

```bash
git checkout main
git pull
git checkout -b fix-divide-by-zero
# 修改代码...
git diff
git add calculator.py
git commit -m "Fix: divide by zero raises ValueError"
git push -u origin fix-divide-by-zero
# 在 GitHub 上创建 PR
# PR 合并后清理
git checkout main
git pull
git branch -d fix-divide-by-zero
```

## 3. Python 测试基础

### 项目结构

```
项目目录/
├── calculator.py              # 被测代码
├── test_calculator.py         # 测试文件
├── requirements.txt           # 依赖
└── pyproject.toml             # 项目配置
```

### 被测代码示例

```python
def add(a, b):
    return a + b

def divide(a, b):
    if b == 0:
        raise ValueError("Cannot divide by zero")
    return a / b
```

### 测试文件示例

```python
import pytest
from calculator import add, divide

class TestAdd:
    def test_positive_numbers(self):
        assert add(2, 3) == 5

    def test_negative_numbers(self):
        assert add(-1, -1) == -2

class TestDivide:
    def test_basic(self):
        assert divide(10, 2) == 5

    def test_divide_by_zero(self):
        with pytest.raises(ValueError, match="Cannot divide by zero"):
            divide(1, 0)
```

### 运行测试

```bash
pytest test_calculator.py -v   # 运行测试，-v 显示详细输出
```

### 输出示例

```
test_calculator.py::TestAdd::test_positive_numbers PASSED
test_calculator.py::TestAdd::test_negative_numbers PASSED
test_calculator.py::TestDivide::test_divide_by_zero PASSED
======================== 10 passed =========================
```

## 4. GitHub Actions 详解

### 配置文件位置

```
.github/workflows/ci.yml
```

### 核心概念

| 概念 | 含义 |
|------|------|
| `name` | 工作流名称，显示在 GitHub 页面上 |
| `on` | 触发条件，什么时候运行 |
| `jobs` | 任务集合 |
| `runs-on` | 用什么机器运行（如 ubuntu-latest） |
| `steps` | 具体执行步骤，按顺序执行 |
| `strategy.matrix` | 矩阵策略，并行测试多个环境 |

### 完整 CI/CD 配置

```yaml
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  # CI：测试
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ["3.10", "3.11", "3.12"]

    steps:
      - uses: actions/checkout@v4

      - name: Set up Python ${{ matrix.python-version }}
        uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Run tests
        run: pytest test_calculator.py -v

  # CD：打包
  build:
    needs: test
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install build tools
        run: pip install build

      - name: Build package
        run: python -m build

      - name: Upload build artifacts
        uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/

  # CD：发布到 PyPI
  publish:
    needs: build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'

    steps:
      - uses: actions/checkout@v4

      - name: Download build artifacts
        uses: actions/download-artifact@v4
        with:
          name: dist
          path: dist/

      - name: Publish to PyPI
        if: ${{ secrets.PYPI_API_TOKEN != '' }}
        uses: pypa/gh-action-pypi-publish@release/v1
        with:
          password: ${{ secrets.PYPI_API_TOKEN }}
```

### 关键配置解释

**`needs` — 任务依赖：** test 不过，build 不会执行。

**`matrix` — 矩阵测试：** 同时启动 3 台虚拟机，分别用 3 个 Python 版本跑测试。

**`if` — 条件执行：**
- `github.ref == 'refs/heads/main'` — 只在 main 分支执行
- `github.event_name == 'push'` — 只在 push 时执行（PR 不触发发布）

**`secrets` — 密钥管理：** Token 存在 GitHub 仓库 Settings → Secrets 里，代码中通过 `${{ secrets.XXX }}` 引用，不会泄露。

**`artifact` — 构建产物传递：** build 任务上传文件，publish 任务下载文件，两个任务在不同虚拟机上通过 artifact 传递数据。

## 5. CI/CD 具体步骤

### 流水线可视化

```
开发者 push 代码
       │
       ▼
┌──────────────────────────────┐
│  Job 1: test（CI）             │
│  在 3 个 Python 版本上并行测试   │
└──────────────┬───────────────┘
               │ 全部通过
               ▼
┌──────────────────────────────┐
│  Job 2: build（CD - 打包）      │
│  python -m build              │
│  生成 .whl 和 .tar.gz          │
│  上传为 GitHub Artifact        │
└──────────────┬───────────────┘
               │ 打包成功
               ▼
┌──────────────────────────────┐
│  Job 3: publish（CD - 发布）    │
│  仅在 push main 时触发          │
│  有 token → 发布到 PyPI         │
│  没 token → 跳过，打印提示       │
└──────────────────────────────┘
```

### 每个步骤详解

| 步骤 | 做什么 | 对应配置 |
|------|--------|----------|
| 拉取代码 | 把 GitHub 上的代码下载到虚拟机 | `actions/checkout@v4` |
| 装环境 | 在虚拟机上安装 Python | `actions/setup-python@v5` |
| 装依赖 | `pip install -r requirements.txt` | `run: pip install ...` |
| 跑测试 | `pytest test_calculator.py -v` | `run: pytest ...` |
| 打包 | `python -m build` | `run: python -m build` |
| 上传产物 | 把 .whl 和 .tar.gz 存起来 | `upload-artifact@v4` |
| 发布 | 上传到 PyPI | `pypa/gh-action-pypi-publish` |

## 6. CD 阶段：打包与发布

### 打包配置（pyproject.toml）

```toml
[build-system]
requires = ["setuptools>=68.0", "wheel"]
build-backend = "setuptools.build_meta"

[project]
name = "python-ci-demo"
version = "0.1.0"
description = "A simple calculator for learning CI/CD"
requires-python = ">=3.10"
```

### 打包命令

```bash
pip install build
python -m build
```

生成两个文件：
- `python_ci_demo-0.1.0.tar.gz` — 源码包
- `python_ci_demo-0.1.0-py3-none-any.whl` — 二进制包

### .whl vs .tar.gz

| | .tar.gz（源码包） | .whl（Wheel 包） |
|---|---|---|
| 内容 | 源代码 | 构建好的文件 |
| 安装速度 | 较慢（需要编译） | 较快（直接复制） |
| 类比 | 食材 + 菜谱 | 做好的菜 |

### 文件名含义

```
python_ci_demo-0.1.0-py3-none-any.whl
│              │     │   │    │
│              │     │   │    └── 不依赖特定 CPU 架构
│              │     │   └── 不依赖特定操作系统
│              │     └── 支持 Python 3
│              └── 版本号
└── 项目名
```

### 使用方式

```bash
pip install python_ci_demo-0.1.0-py3-none-any.whl

# 然后在代码中
from calculator import add
print(add(2, 3))  # 5
```

### PyPI 是什么

PyPI = Python Package Index，Python 官方软件包仓库（https://pypi.org）

`pip install xxx` 就是从 PyPI 下载的。

| | GitHub | PyPI |
|---|---|---|
| 存什么 | 源代码 | 打包好的软件 |
| 谁用 | 开发者（看代码、改代码） | 用户（装了直接用） |
| 安装方式 | `git clone` | `pip install` |

### 发布到 PyPI 的要求

- 注册免费，需要邮箱验证
- 包名全网唯一，冲突就发布失败
- 同一版本号不能重复上传
- 需要创建 API Token（https://pypi.org/manage/account/token/）

### 不同场景的 CD 做法

| 场景 | 部署方式 |
|------|----------|
| Python 库 | 发布到 PyPI |
| Web 应用 | SSH 到服务器执行部署脚本 |
| Docker 应用 | 构建镜像推送到 Docker Hub |
| 静态网站 | 部署到 GitHub Pages |

## 7. Bash 命令详解

以这条命令为例：

```bash
cd "C:/Users/bingu/Desktop/ai测试" && pip install setuptools wheel -q && python setup.py sdist bdist_wheel 2>&1 | tail -5
```

拆开来看，一共 5 个知识点。

### 7.1 `&&` — 串联执行

```bash
命令A && 命令B && 命令C
```

A 成功了才执行 B，B 成功了才执行 C。任何一步失败，后面的就不跑了。

对比 `;`（分号）：不管前面成不成功，后面的都会跑。

### 7.2 `""` — 双引号

```bash
cd "C:/Users/bingu/Desktop/ai测试"
```

路径里有中文或特殊字符时，必须用双引号包起来，否则 Bash 会把空格或中文当成命令分隔符。

### 7.3 文件描述符 — `2>&1` 的核心

Linux/Windows 的命令有三个输出通道：

| 编号 | 名称 | 作用 | 默认输出到 |
|------|------|------|-----------|
| 0 | stdin | 输入 | 键盘 |
| 1 | **stdout** | 正常输出 | 屏幕 |
| 2 | **stderr** | 错误输出 | 屏幕 |

虽然默认都显示在屏幕上，但 stdout 和 stderr 是**两条独立的管道**。

### 7.4 `2>&1` — 把错误合并到正常输出

| 部分 | 含义 |
|------|------|
| `2` | 错误输出（stderr） |
| `>&1` | 重定向到通道 1（stdout）所在的地方 |

效果：把 stderr 和 stdout 合并成一条流。

为什么需要？看对比：

```bash
# 没有 2>&1：管道只接 stdout，错误信息直接显示在屏幕，不经过 tail
python setup.py sdist bdist_wheel | tail -5

# 有 2>&1：合并后一起通过管道，tail 能看到所有输出
python setup.py sdist bdist_wheel 2>&1 | tail -5
```

### 7.5 `|` 管道 + `tail -5`

| 符号 | 作用 |
|------|------|
| `\|` | 管道：把前一个命令的输出，作为后一个命令的输入 |
| `tail -5` | 只显示最后 5 行 |

打包命令输出很长（几十行），用 `tail -5` 只看最后几行确认是否成功。

### 完整执行流程

```
cd "C:/Users/bingu/Desktop/ai测试"
        │ 成功
        ▼
pip install setuptools wheel -q    # -q 安静模式，不显示下载过程
        │ 成功
        ▼
python setup.py sdist bdist_wheel
        │
        │ 正常输出 → 通道 1
        │ 错误输出 → 通道 2
        ▼
2>&1   把通道 2 合并到通道 1
        │
        ▼
|      管道，把合并后的输出传给下一个命令
        │
        ▼
tail -5   只取最后 5 行显示
```

### 常见重定向写法

```bash
command > file.txt        # 正常输出写入文件（覆盖）
command >> file.txt       # 正常输出追加到文件
command 2> errors.txt     # 错误输出写入文件
command 2>&1 > all.txt    # 合并后全部写入文件
command > /dev/null 2>&1  # 全部丢弃（静默执行）
```

`/dev/null` 是一个"黑洞"，写进去的东西全部消失，常用于不想看任何输出的场景。

## 9. ADB 基础

### 什么是 ADB

ADB（Android Debug Bridge，安卓调试桥）是一个**命令行工具**，让电脑可以和安卓设备通信，主要用途：

- 安装/卸载 App
- 文件传输
- 查看日志
- 模拟操作（点击、滑动、输入）
- 调试和性能分析
- 自动化测试

### 工作原理

ADB 由 3 部分组成：

```
电脑端                              手机端
┌──────────────┐                ┌──────────────┐
│ adb client   │                │ adbd 守护进程  │
│ (你敲的命令)  │ ──────────────→│ (运行在设备上)  │
└──────┬───────┘                └──────────────┘
       │
       ▼
┌──────────────┐
│ adb server   │
│ (后台服务)    │
└──────────────┘
```

- **client（客户端）：** 你在终端敲的 `adb` 命令
- **server（服务端）：** 后台进程，管理 client 和 device 之间的通信
- **daemon（守护进程）：** 运行在手机上的 `adbd`

### 安装 ADB

**Windows：**
1. 下载 Android Platform Tools：https://developer.android.com/tools/releases/platform-tools
2. 解压到任意目录（如 `C:\platform-tools`）
3. 添加到系统环境变量 PATH
4. 打开 cmd，输入 `adb version` 验证

**Mac：**
```bash
brew install android-platform-tools
```

**Linux：**
```bash
sudo apt install android-tools-adb
```

### 手机准备工作

1. 打开**开发者选项**：设置 → 关于手机 → 连续点击"版本号"7 次
2. 进入开发者选项，打开 **USB 调试**
3. 用数据线连接电脑，手机弹出"允许调试"，点击允许

## 10. 连接设备

### USB 连接

```bash
# 查看已连接的设备
adb devices
```

## 18. 网络调试

### 端口转发（forward）

把**电脑的端口**映射到**手机的端口**，电脑访问本地端口就是访问手机服务。

```bash
# 把电脑 8080 端口转发到手机 8080 端口
adb forward tcp:8080 tcp:8080

# 现在浏览器访问 http://localhost:8080 = 访问手机的 8080

# 查看所有转发
adb forward --list

# 删除某个转发
adb forward --remove tcp:8080

# 删除所有
adb forward --remove-all
```

**应用场景：** 调试手机上的 Web 服务、调试 Chrome DevTools。

### 反向代理（reverse）

把**手机的端口**映射到**电脑的端口**，手机访问本地端口就是访问电脑服务。

```bash
# 把手机 8080 端口反向映射到电脑 8080
adb reverse tcp:8080 tcp:8080

# 现在手机访问 http://localhost:8080 = 访问电脑的 8080
```

**应用场景：** 手机 App 调用本地开发的后端服务、电脑无 WiFi 时让手机用电脑的网络。

### 抓包工具

| 工具 | 说明 |
|------|------|
| **Charles** | 收费，界面友好，最常用 |
| **Fiddler** | 免费（Windows） |
| **Wireshark** | 免费，专业，学习曲线陡 |
| **Mitmproxy** | 命令行工具 |

抓包基本流程：
1. 电脑装抓包工具，开启代理
2. 手机 WiFi 设置代理为电脑 IP + 端口
3. 手机安装抓包工具的证书（HTTPS 必须）
4. 操作 App，观察抓到的请求

## 20. 完整测试工作流示例

### 场景：测试一个登录功能的稳定性

下面是一个完整的 Bash 测试脚本：

```bash
#!/bin/bash
# test_login.sh - 登录功能稳定性测试

PACKAGE="com.example.app"
ACTIVITY="$PACKAGE/.LoginActivity"
LOG_DIR="./test_logs"
DATE=$(date +%Y%m%d_%H%M%S)

# 1. 准备测试目录
mkdir -p "$LOG_DIR"

# 2. 检查设备连接
echo "===== 检查设备 ====="
adb devices

# 3. 清理 App 数据，确保从干净状态开始
echo "===== 清理 App 数据 ====="
adb shell pm clear $PACKAGE

# 4. 清空日志
adb logcat -c

# 5. 开始记录日志（后台）
adb logcat > "$LOG_DIR/logcat_$DATE.txt" &
LOGCAT_PID=$!

# 6. 启动 App
echo "===== 启动 App ====="
adb shell am start -W -n $ACTIVITY

# 7. 等待启动完成
sleep 3

# 8. 截图（启动后）
adb shell screencap -p /sdcard/login_screen.png
adb pull /sdcard/login_screen.png "$LOG_DIR/login_screen_$DATE.png"

# 9. 模拟登录操作
echo "===== 模拟登录 ====="
adb shell input tap 500 800              # 点击用户名输入框
adb shell input text "testuser"          # 输入用户名
adb shell input tap 500 1000             # 点击密码输入框
adb shell input text "password123"       # 输入密码
adb shell input tap 500 1200             # 点击登录按钮

# 10. 等待响应
sleep 5

# 11. 截图（登录后）
adb shell screencap -p /sdcard/after_login.png
adb pull /sdcard/after_login.png "$LOG_DIR/after_login_$DATE.png"

# 12. 跑 Monkey 测试 200 个随机事件
echo "===== Monkey 压力测试 ====="
adb shell monkey -p $PACKAGE --throttle 500 -v 200 > "$LOG_DIR/monkey_$DATE.txt"

# 13. 收集性能数据
echo "===== 收集性能数据 ====="
adb shell dumpsys meminfo $PACKAGE > "$LOG_DIR/meminfo_$DATE.txt"
adb shell dumpsys gfxinfo $PACKAGE > "$LOG_DIR/gfxinfo_$DATE.txt"

# 14. 停止日志记录
kill $LOGCAT_PID

# 15. 检查是否有 Crash
echo "===== 检查 Crash ====="
if grep -q "FATAL EXCEPTION" "$LOG_DIR/logcat_$DATE.txt"; then
    echo "❌ 发现 Crash！"
    grep -A 20 "FATAL EXCEPTION" "$LOG_DIR/logcat_$DATE.txt"
    exit 1
else
    echo "✅ 测试通过，无 Crash"
fi

echo "===== 测试完成，结果保存在 $LOG_DIR ====="
```

### 执行方式

```bash
chmod +x test_login.sh
./test_login.sh
```
